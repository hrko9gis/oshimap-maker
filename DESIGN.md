## 1. 目的とスコープ

### 1.1 目的

計画書v2 §4 の目的を継承しつつ、Maker（作成者向けツール）に限定して以下を目的とする。

1. ファンや地域団体が、専門知識なしで推し活マップ（アニメ聖地巡礼・町歩き・スタンプラリー
   マップ）のデータを作成・下書き・公開できるようにする
2. 既存の公開アプリ「たまゆら竹原 聖地さんぽマップ」で確立した権利配慮型・データ仕様
  （`data/schema.md`）をそのまま作成者向けフォームの入力仕様として再利用する
3. Maker が生成したデータ・設定を、ビューア（公開マップ）側にソースコードを混在させずに
   受け渡せる方式を定義する
4. v1着手（フォーム・Supabase等の実装）の判断材料となる、技術構成・データモデル・
   画面構成・機能仕様を、Go条件成立後すぐに着手できる粒度で明文化する

### 1.2 実装する（本書がスコープとして定義するもの）

計画書v2 §9〜16 のうち、v0で実装しなかった「作成者向けレイヤー」を対象とする。

- プロジェクト作成・プロジェクト設定
- スポット登録フォーム（`data/schema.md` のプロパティスキーマに準拠）
- 地図クリックによる位置登録・ピンドラッグ修正
- 下書き（draft）／レビュー中（review）／公開（published）のステータス管理
- 公開前チェック（権利・安全チェックリスト）
- 自動生成マップ（＝既存ビューアの機能。Makerは新規実装しない。データを渡すのみ）
- スタンプラリー設定（合言葉ハッシュ生成・スタンプ対象ON/OFF）
- データ出力（GeoJSON／CSV）
- Maker → ビューアへのデータ・設定の受け渡し（エクスポート／デプロイ）

### 1.3 実装しない（明示的非スコープ。v0設計書 §3.2 および CLAUDE.md を継承）

- QRコードの物理設置を前提とするチェックイン（施設交渉が必要なため）
- GPS厳密判定によるチェックイン、ルート最適化
- コメント欄、写真投稿、SNS的なタイムライン
- 動的OGP（プロジェクトごとのOGP画像動的生成）
- 画像アップロード機能（作品画像・スクリーンショットの掲載は恒久的に非スコープ。
  自分で撮影した風景写真等の掲載可否はv0設計書と同様に将来検討事項のままとし、
  本書でも着手しない）
- 認証方式の詳細設計（ログイン・パスワードリセット等の実装仕様）。本書ではロール
  モデル（`project_members`）とアクセス制御の要件のみを扱い、認証プロバイダの実装
  詳細は着手時に別途設計する
- 本書に基づく実装作業そのもの（§0参照。Go条件成立まで着手しない）

## 2. 前提：フェーズ1相当は実装済み

計画書v2 §16「開発ステップ」の現状メモの通り、**フェーズ1相当（公開用Webマップ）は
既存の公開アプリ「たまゆら竹原 聖地さんぽマップ」として実装・動作確認済み**である
（本リポジトリ `oshimap` の `src/`・`data/`）。

本書では、このアプリを**「生成される公開マップ（ビューア）」のリファレンス実装**として扱う。
すなわち：

- Maker が将来生成するデータ・設定は、このアプリが読み込める形式（`spots.geojson` の
  プロパティスキーマ、`stamp_answers.json` のハッシュ形式）と**互換**でなければならない
- ビューア側の表示機能（カテゴリ色分け・凡例、モデルコース番号表示、前へ/次へツアー、
  背景地図切替、チェックイン済み表示、選択スポット強調、日英切替）は、このリファレンス
  実装が持つ機能仕様そのものを指す。Maker はこれらを再実装しない
- 実装済みの技術的事実（例：`sort_order` が1始まりのモデルコース順であること、
  カテゴリenumが `anime_spot | townscape | transport | rest | shopping | viewpoint` の
  6値であること、`location_accuracy` が `exact | approximate | area` の3値であること）は
  Makerのデータモデル・フォーム仕様の一次情報として扱う

## 3. アーキテクチャ（必須方針）

### 3.1 基本方針：ソースコード分離

**既存アプリ（ビューア）と Maker（作成ツール）のソースコードは分離する。** 両者を
統合した単一アプリケーション（単一デプロイ単位）にはしない。理由は次の2点。

1. **横展開の独立性（v0設計書 §6 継承）**：ビューアは「リポジトリをcloneして
   `data/spots.geojson` を差し替えれば別地域マップになる」構造を保つ。ビューアの
   リポジトリに Maker のコード（フォーム・保存・認証・管理画面等）が混在すると、
   clone して展開する作成者・地域団体にとって不要な複雑さと依存関係（Supabase等）を
   持ち込むことになり、この横展開容易性を損なう
2. **撤退時の独立性（v0設計書 §5 継承）**：v0の撤退基準に該当した場合、
   「たまゆら竹原 聖地さんぽマップ単体を静的サイトとして維持する」ことが決定済みの
   方針である。Maker 側の開発が停止・破棄されても、ビューア単体が今まで通り
   ビルド・デプロイ可能な状態を維持できなければならない。ソースを分離しておくことで、
   Maker の運命に関わらずビューアの独立運用が保証される

### 3.2 リポジトリ構成の選択：別リポジトリ方式（推奨・既定）

「別リポジトリ」と「モノレポの別パッケージ」の両案を検討し、以下の理由で
**別リポジトリ方式を既定**とする。

| 観点 | 別リポジトリ（採用） | モノレポ別パッケージ |
|---|---|---|
| 撤退時の独立性 | 本リポジトリ(`oshimap`)は一切変更不要。現状のまま静的サイトとして凍結できる | Makerパッケージの追加自体が本リポジトリへの変更となり、「たまゆら単体」の純粋さが損なわれる |
| clone&差替えの単純さ | 既存の「clone して `data/` を差し替え」フローに変更なし | ワークスペース定義・ビルド設定ごとcloneする必要があり、複雑化 |
| デプロイ単位 | ビューア＝静的サイト（Cloudflare Pages等）、Maker＝別デプロイ（Supabase等を伴う動的アプリ）と、実態に即して自然に分離 | 単一リポジトリでも各パッケージは別デプロイになり、分離の利点が薄い一方で管理コストは増える |
| 開発初速 | 若干のコード重複（型定義など）が発生しうる | 共有コードの参照は容易 |

結論：**本リポジトリ（`oshimap`）はビューアのリファレンス実装として現状のまま維持し、
Maker は新規リポジトリ（例：`oshimap-maker`）として開発する。** モノレポ化は、
将来的に共有コードの重複が保守負担として顕在化した場合にのみ再検討する
（YAGNI。現時点でビューアの実装は1インスタンスのみであり、抽象化を先取りしない）。

### 3.3 共有すべき契約と共有方法

以下は「ビューアが読み取れるデータ・設定の形」という契約であり、Maker とビューアの
両方が一致していなければならない。ソースコードは統合しないが、**契約は明文化して
同期する**。

| 共有対象 | 内容 | 共有方法（既定） |
|---|---|---|
| GeoJSON プロパティスキーマ | `data/schema.md` の `SpotProperties`（id, title, category, summary, source_url, location_accuracy, stamp_enabled, sort_order 等） | 本書 §5.2 に転記し、双方のリポジトリの実装がこの仕様に追従する。型定義（TypeScript）は当面**各リポジトリで個別に保持**し、スキーマ変更時は本書を更新してから両リポジトリに反映する |
| 合言葉ハッシュ方式 | NFKC正規化＋前後空白除去＋SHA-256(hex)（`src/lib/stampHash.ts` 準拠） | 同上。Maker側でもハッシュ生成時に同一アルゴリズムを実装する（Node.js `crypto` モジュールで再現可能） |
| コースナビゲーション仕様 | `sort_order` 昇順＝モデルコース順（`src/lib/course.ts` 準拠） | 本書に仕様を転記。Maker はプレビュー機能で同じ並び替えロジックを再実装する |
| 背景地図スタイル定義 | 地理院タイル／OpenFreeMapのURL・言語別既定（`src/lib/mapStyles.ts` 準拠） | Maker のプレビュー地図でも同一の背景地図定義を用いる（プレビューの見た目をビューアと一致させるため） |
| i18n辞書構造 | `ja.json` / `en.json` のキー構造 | Maker のフォームは `Bilingual`（`{ ja, en }`）型の入力欄を前提に設計する |

**共有パッケージへの切り出しは「必要に応じて」**（GOAL契約の通り）行う。具体的には、
上記の重複保守コストが実際に問題化した時点（例：スキーマ変更のたびに両リポジトリで
食い違いバグが発生する等）で、`@oshimap/shared`（型定義・スキーマ検証関数・ハッシュ関数・
コースナビ関数・背景地図定義・i18n型）を npm パッケージ（公開 or GitHub Packages）として
切り出し、両リポジトリが依存する。これは2アプリの**ソースコードの統合ではない**
（共有パッケージはあくまで型・純粋関数のみを含み、UIやアプリロジックを含まない）。

### 3.4 Maker → ビューアのデータ・設定の受け渡し方式（エクスポート／デプロイ）

Maker で作成・編集したプロジェクトを、公開マップ（ビューアのインスタンス）として
世に出すための受け渡し方式を、**最小構成**と**推奨構成**の2段階で定義する。

#### 3.4.1 受け渡しの単位：「配布バンドル」

Maker は、1プロジェクトにつき以下のファイル一式（配布バンドル）を生成する。
これは現行ビューアの `data/` ディレクトリ構成をそのまま踏襲する。

```
export/
├── spots.geojson       # data/schema.md 準拠。status=published のスポットのみ含む
├── stamp_answers.json  # 合言葉のSHA-256ハッシュのみ（平文を含まない）
└── project.json        # 新規：プロジェクト（マップ）単位の設定
```

`project.json` は、現行ビューアにはまだ存在しない**プロジェクト設定ファイル**であり、
計画書v2 §13.1 の `projects` テーブルの主要カラムをエクスポートしたものである。

```jsonc
{
  "title": { "ja": "たまゆら竹原 聖地さんぽマップ", "en": "Tamayura Takehara Pilgrimage Map" },
  "area_name": { "ja": "広島県竹原市", "en": "Takehara, Hiroshima" },
  "description": { "ja": "...", "en": "..." },
  "default_language": "ja",
  "license": "CC BY 4.0",
  "disclaimer": { "ja": "非公式ファンプロジェクトです。...", "en": "..." }
}
```

ビューア側がこの `project.json` を読み込んで表示文言（タイトル・注意事項ページ等）に
反映する機能は、現行のビューア実装にはまだ存在しない（現行実装はハードコードされた
`ja.json`/`en.json` と `AboutPage.tsx` の固定文を用いる）。この対応は
**ビューアの汎用化作業（v1、Go条件成立後の開発ステップの一部）**として扱い、
本書の設計対象に含める（§6・§9参照）が、既存の `oshimap` リポジトリを今すぐ変更する
ものではない。

#### 3.4.2 最小構成（初期実装・既定）

1. 作成者が Maker 上でプロジェクトを作成・スポット登録・下書き→公開ステータス変更を行う
2. Maker が「配布バンドルをダウンロード」操作で上記3ファイルのZIPを生成する
3. 作成者は、ビューアのテンプレートリポジトリ（`oshimap` を汎用化したテンプレート、
   または `oshimap` そのものを clone したもの）に対して、ダウンロードしたファイルで
   `data/` 配下を差し替え、`git commit` & `push` する
4. 既存のホスティング（Cloudflare Pages / GitHub Pages）の「mainマージで自動デプロイ」
   の仕組みがそのまま働き、公開される

この方式は、v0設計書 §6 の「リポジトリをcloneしてGeoJSONを差し替えれば別地域マップに
なる」構造をそのまま活かすものであり、追加インフラを必要としない。**Makerの最小構成
（計画書v2 §12.2：ローカルJSON保存・認証なし）と自然に対応する。**

#### 3.4.3 推奨構成（v1・Supabase構成時）

Maker がSupabase等でホスティングされる場合、配布バンドルの生成・引き渡しを自動化する。

1. Maker の Edge Function（または同等のサーバーサイド処理）が、公開操作をトリガーに
   配布バンドルを生成する
2. 生成したバンドルを、GitHub API を用いてビューア用リポジトリ（作成者が指定した
   フォーク／新規リポジトリ）に対しコミット（またはプルリクエスト）として送信する
3. ビューア用リポジトリ側のCI（既存のCloudflare Pages連携）が自動デプロイする

この方式でも、**Maker はビューアのソースコードを直接変更・デプロイしない**。
あくまで `data/` 配下のファイルをコミットするのみであり、ビューアのビルド・
デプロイパイプラインの所有権はビューア側リポジトリに残る。これにより3.1の
ソース分離方針を維持したまま自動化を実現する。

## 4. 技術構成

計画書v2 §12 を継承し、既存ビューア実装（フェーズ1）との整合を明記する。

### 4.1 ビューア側（公開マップ）：既存構成の再利用

既存アプリ（本リポジトリ）の構成をそのまま用いる。新規実装は行わない。

| 役割 | 技術 | 備考 |
|---|---|---|
| フロントエンド | React 18 + Vite + TypeScript | 実装済み |
| 地図 | MapLibre GL JS | 実装済み |
| 背景地図 | 地理院タイル（淡色・写真）／OpenFreeMap（OSMベクトル） | 言語別既定・切替実装済み（`src/lib/mapStyles.ts`） |
| UI | Tailwind CSS | 実装済み |
| データ | ビルド時に取り込む `data/spots.geojson`・`data/stamp_answers.json` | Makerが生成する配布バンドルと同一スキーマ |
| スタンプ保存 | localStorage（端末内） | 実装済み。Maker側の `stamp_logs`（サーバー保存）とは独立した仕組みとして維持する |
| i18n | 辞書JSON（ja/en） | 実装済み。§3.4.1 の `project.json` 対応は将来の汎用化タスク |
| ホスティング | Cloudflare Pages（第一候補）／GitHub Pages | mainマージで自動デプロイ |

### 4.2 作成者側（Maker）：最小構成と推奨構成

計画書v2 §12.1/12.2 を踏襲する。

| 役割 | 最小構成（MVP） | 推奨構成（v1本格版） |
|---|---|---|
| フロントエンド | React / Vite / TypeScript | 同左 |
| 地図表示（登録・プレビュー用） | MapLibre GL JS（ビューアと同一ライブラリで見た目を揃える） | 同左 |
| UI | Tailwind CSS | 同左 |
| データ保存 | ローカルJSON（ブラウザ内、IndexedDBまたはファイル書き出し） | Supabase（PostgreSQL + PostGIS） |
| 認証 | なし（単一作成者・端末内完結） | Supabase Auth（プロジェクト単位の共同編集を想定） |
| API | なし（クライアント内処理のみ） | Supabase Edge Functions |
| 位置検索補助 | 地図クリックのみ | 地図クリック＋住所検索（ジオコーディングAPI連携） |
| デプロイ先受け渡し | §3.4.2 手動エクスポート | §3.4.3 自動エクスポート（GitHub API連携） |
| ホスティング（Maker自体） | Vercel / Netlify / Cloudflare Pages（静的+クライアント処理） | 同左＋Supabaseプロジェクト |

**既定は最小構成とする。** 最小構成でもフェーズ2（スポット登録機能）の主要価値
（フォーム入力→GeoJSON生成→ビューアへ反映）は検証可能であり、Supabase等の
インフラ投資はGo条件成立後、実際の複数作成者・共同編集ニーズが確認されてから
判断する（YAGNI。計画書v2 §12.3 が推奨する「React + MapLibre + Supabase/PostGIS」は
v1の目標構成として本書にも記載するが、MVPの必須要件にはしない）。

## 5. データモデル

計画書v2 §13 の4テーブルを、実装済みのビューア側スキーマ（`data/schema.md`）と
整合させて再定義する。

### 5.1 projects テーブル

| カラム | 内容 | ビューア側との対応 |
|---|---|---|
| id | プロジェクトID | - |
| title | マップ名（`{ ja, en }`） | `project.json.title`（§3.4.1）として配布 |
| slug | URL用ID | 配布バンドルの出力先ディレクトリ名等に利用 |
| description | 説明文（`{ ja, en }`） | `project.json.description` |
| theme_type | アニメ、ゲーム、VTuberなど | Maker管理画面の分類用途のみ。ビューアには渡さない |
| area_name | 対象地域（`{ ja, en }`） | `project.json.area_name` |

## 7. 機能仕様

計画書v2 §9 を踏襲し、実装済み機能（ビューア）とMaker新規機能を区別する。

### 7.1 プロジェクト作成機能（Maker・新規）

計画書v2 §9.1 の入力項目（マップ名・対象作品・対象地域・説明文・公開範囲・対応言語・
ライセンス・注意書き・公式サイトURL）をフォームとして実装する。保存先は§4.2の
最小構成ではローカルJSON、推奨構成では `projects` テーブル。

### 7.2 スポット登録フォーム（Maker・新規）

計画書v2 §9.2 の入力項目を、§5.2 のデータモデルに従ってフォーム化する。

- 必須項目（`data/schema.md` 準拠）：id、title(ja/en)、category、summary(ja/en、各120字以内)、
  source_url、source_name(ja/en)、location_accuracy、stamp_enabled、sort_order、
  visit_difficulty、status
- `stamp_enabled=true` の場合、`stamp_keyword_hint`(ja/en) と `stamp_keyword_answer`（平文、
  Maker内部のみ保持）を必須とする
- `summary` は120字を超える入力をフォームレベルで警告する（ビューアと同じ品質基準を
  作成時点で担保する）
- カテゴリ・位置精度・訪問難易度は既存enumのみを選択肢とし、自由入力を許可しない
  （enumの拡張は本書のスコープ外。将来テンプレート機能で拡張する場合は別途設計する）

### 7.3 地図上での位置登録・編集（Maker・新規）

計画書v2 §9.3 を踏襲する。

- 地図クリックで緯度経度を登録し、`location_accuracy` の選択を促す
- ピンをドラッグして位置修正
- 施設名・住所検索（推奨構成でのみ、ジオコーディングAPIを利用）
- 背景地図切替（ビューアと同一の地理院タイル／OpenFreeMap定義を使用し、位置確認時の
  見た目をビューアと一致させる）
- 住宅地・私有地近接時は `location_accuracy: "area"` を推奨する旨をUIに明記する
  （CLAUDE.md制約#5・v0設計書§7.3の座標取得ルールを継承。Google Mapsからの座標コピーは
  フォームの注意書きで明示的に禁止する）

### 7.4 下書き／公開ステータス管理（Maker・新規）

`status: draft | review | published` の3段階を実装する。

- draft：作成者のみ閲覧・編集可（推奨構成では`project_members`のeditor以上）
- review：レビュー担当者（`project_members.role = reviewer`）が§8.3のチェックリストを
  確認する状態
- published：配布バンドルのエクスポート対象となる状態。エクスポート後、ビューアに
  反映されて初めて一般公開される（Maker上で`published`にしただけでは一般公開されない。
  §3.4のエクスポート操作が実際の公開トリガーとなる）

### 7.5 自動生成マップ（ビューア・実装済み。Makerは再実装しない）

計画書v2 §9.4 に列挙された機能は、既存ビューアで実装済みである。

- カテゴリ別ピン表示（色分け＋凡例）
- スポット番号（＝モデルコース順、`sort_order`）表示
- スタンプ対象地点のみ表示、チェックイン済み表示
- 選択スポットの強調（ピン拡大＋一覧ハイライト）
- スポット一覧（モデルコース順）、詳細カード（コース位置「N / 総数」表示）
- モデルコース：前へ/次へによるツアー（ゆっくりした移動＋ズーム、`FlyMode: 'tour'`）
- 背景地図切替（言語別既定・言語切替時の写真維持ロジック含む）
- 現在地表示、公式・観光リンク表示、注意事項表示
- 多言語切替（ja/en。UI文言・スポットデータ・地図注記）
- 共有URL（固定OGP）

計画書v2にある「作品別フィルタ」「推し対象別フィルタ」は、現行ビューアの
`FilterChips.tsx` がカテゴリフィルタのみを実装している。作品別・推し対象別フィルタの
追加は、ビューアの汎用化（§9のフェーズ2以降）で検討する拡張項目とし、本書では
Makerの新規実装対象に含めない（ビューア側の変更が必要なため）。

### 7.6 スタンプラリー機能

計画書v2 §9.5・v0設計書 §9 を踏襲する。**方式は実装済みの「合言葉（SHA-256照合）＋
手動チェックイン」を正とする。**

- Maker側：スタンプ対象ON/OFF設定、合言葉の入力→ハッシュ化（§3.3のアルゴリズム）→
  `stamp_answers.json` への格納。**合言葉の平文はMaker内部保存にのみ許可し、
  エクスポート・バージョン管理には含めない**（CLAUDE.md制約#6を継承）
- ビューア側：達成状況・進捗バー・完走バッジ・端末内保存（localStorage）は実装済みの
  まま変更しない
- QRコード物理設置・GPSチェックインは非スコープ（§1.3）

### 7.7 データ出力機能

計画書v2 §9.6 のうち、GeoJSON・CSVはビューア側で実装済み（`AboutPage.tsx` の
ダウンロードボタン、`src/lib/csv.ts`）。Maker側では、作成中データに対して同等の
エクスポート（プレビュー用GeoJSON/CSV、および§3.4の配布バンドル）を提供する。
QRコード一覧・簡易マップ画像/PDF出力は非スコープ（§1.3、後期検討）。

### 7.8 公開前チェック機能（Maker・新規）

計画書v2 §9.7 のチェック項目を、`review`→`published` 遷移時のチェックリストUIとして
実装する。

- 公式画像・アニメ画像・無断転載写真を使用していないか（テキストのみで確認、自動検知は行わない）
- 説明文（`summary`）が独自作成の短い概要か（120字制限で機械的にチェック）
- 公式・観光サイトへのリンク（`source_url`）が設定されているか（必須項目のためフォーム側で担保）
- 私有地・住宅地への立入を促していないか（`location_accuracy`が`area`かどうかの確認を促す）
- 営業時間・休館日は公式サイト確認を促す文言になっているか
- 非公式ファンプロジェクトである旨が `projects.disclaimer` に含まれているか

全項目を満たさない限り `published` への遷移を許可しない（フォームバリデーションと
チェックボックスの併用）。

## 8. 権利・安全

計画書v2 §11・v0設計書 §10・CLAUDE.mdの制約をMakerにもそのまま継承する。

### 8.1 基本方針（継承）

- 公式・観光サイトの本文・画像を転載せず、独自に作成した短い概要文と公式・観光サイトへの
  リンクを掲載する
- 作品画像・アニメ本編のスクリーンショット・公式サイト掲載写真・ファンアート等は
  原則として掲載しない。Makerのフォームには画像アップロード欄自体を設けない（§1.3）
- 公式が明言していない場所を「モデル地」と断定する表現を使わない（フォームのヘルプ
  テキストで「作品ゆかり」「ファンに親しまれる」等の言い回しを推奨例として提示する）
- 座標はGoogle Mapsから機械的に取得しない（§7.3）

### 8.2 Maker固有の安全設計

- **合言葉の平文管理**：`stamp_keyword_answer` はMaker内部のデータストア（ローカルJSON
  またはSupabaseの管理者専用テーブル/カラム）にのみ保持し、エクスポート・バージョン管理・
  クライアントに露出するAPIレスポンスのいずれにも平文を含めない。エクスポート時は
  必ずハッシュ化を経由する（§3.3・§5.2）
- **下書きの非公開性**：`status: draft | review` のスポットはエクスポート対象外とし、
  ビューアに一切渡さない（意図しない下書き公開の防止）
- **公開範囲の担保**：`visibility: private | unlisted` のプロジェクトは配布バンドルの
  生成・エクスポート操作自体を無効化する（`public` のみエクスポート可能とする）

### 8.3 命名・表現の制約（継承・必須）

- 「推し旅」を名称・UI文言・ドキュメント・フォームのプレースホルダ等いかなる箇所にも
  使用しない（§0参照）
- 「モデル地」と断定する表現を、Makerのフォームのラベル・ヘルプテキスト・自動生成文言に
  使用しない
- 権利・安全に関するチェックリスト文言は、v0設計書 §10・計画書v2 §11.4 の注意書き文面を
  そのまま `projects.disclaimer` の初期値（テンプレート文）として提供する

## 10. リポジトリ構成案・Claude Code への委任方針

### 10.1 リポジトリ構成案

§3.2の方針に基づき、以下の2リポジトリ構成とする。

```
oshimap/                      # 既存リポジトリ（変更しない）。ビューアのリファレンス実装
├── data/                     # spots.geojson, stamp_answers.json, schema.md
├── src/                      # 既存ビューア実装
├── scripts/                  # validate-data.mjs, check-links.mjs
└── docs/ideas/               # 計画書・v0設計書・本書

oshimap-maker/                 # 新規リポジトリ（Go条件成立後に作成）
├── data/                      # ローカルJSON保存時のプロジェクトデータ（最小構成）
├── src/
│   ├── components/            # ProjectForm, SpotForm, MapPicker, PreviewMap, ...
│   ├── pages/                 # Dashboard, ProjectSettings, SpotList, SpotEditor, Preview, Export
│   ├── lib/
│   │   ├── schema/             # §5・§3.3の契約を実装（型・検証・ハッシュ・コースナビ・背景地図定義）
│   │   ├── export/              # 配布バンドル生成（§3.4）
│   │   └── storage/             # ローカルJSON or Supabaseクライアント
│   └── main.tsx
├── supabase/                  # 推奨構成移行時：migrations, edge functions（v1）
├── README.md                  # 本書へのリンク、セットアップ手順
└── DESIGN.md                  # 本書（`oshimap_maker_design.md`）のコピーを配置する運用とする
```

`oshimap-maker` の `DESIGN.md` は、本リポジトリの `CLAUDE.md` が `docs/ideas/oshimap_v0_design.md`
を参照する運用（既存の実運用パターン）を踏襲し、本書を一次情報源として参照させる。

### 10.2 Claude Code への委任方針

- `oshimap-maker` リポジトリの `CLAUDE.md` には、本書 §1.3・§8 の非スコープ・命名・
  権利安全制約をそのまま転記する
- フェーズ2（§9.2）を1セッション単位の委任範囲の目安とし、各フェーズ完了時に
  「配布バンドルが§3.4.1の形式で生成され、既存ビューア（`oshimap`）にそのまま
  取り込んで表示確認できること」をレビューゲートとする
- 本書の変更（データモデル・契約の変更）が生じた場合は、§3.3の共有契約の同期手順に
  従い、`oshimap` 側の `data/schema.md` との整合を都度確認する
- Go条件が成立するまでは、`oshimap-maker` リポジトリの作成・実装作業そのものに
  着手しない（§0・§9.1）
